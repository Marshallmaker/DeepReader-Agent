# DeepReader Agent 项目开发说明书

## 1. 项目概述

**项目名称**：DeepReader Agent  
**目标**：开发一个 Web 应用，支持用户登录/注册，能够批量上传 PDF 研报/论文，利用国产大模型自动精准提取核心指标（如营收、增长率、样本量等），并输出对比表格（Excel 或 Markdown）。同时集成基于硅基流动平台的 AI 聊天问答小组件，提供基于当前报表内容的辅助解读能力。  
**适用场景**：金融研报分析、学术论文批量对比、数据调研团队。  
**技术约束**：前端采用 React 18，后端采用 Python FastAPI，数据库使用 MySQL 8.0.30，大模型通过硅基流动平台统一调用 DeepSeek-V3 统一接口，任务异步处理采用 Celery + Redis。

## 3. 功能模块、业务边界与设计细节

### 3.1 用户与权限管理模块

- **明确目标**：建立基于角色访问控制（RBAC）的身份隔离体系,支持 Token 双重刷新机制。普通用户仅能查看和对比自己上传的数据；管理员拥有全局最高权限，可穿透查看所有用户的数据。
- **具体功能规范**：

- 注册：普通用户通过邮箱、密码注册。系统对邮箱格式进行严格正则校验，密码在后端采用 `passlib (bcrypt)` 进行单向哈希加密存储。新注册账号默认处于激活状态。本版本暂不引入第三方验证码，采用防刷限流机制限制单 IP 注册频率（每分钟最大 3 次）。*管理员账号不提供开放注册通道，仅能通过数据库初始脚本注入。*
- **登录与鉴权（双 Token 策略）**：
  - `access_token`：有效期 15 分钟，载荷中包含 `user_id`、`email`、`is_admin`。**前端安全存储策略**：由 Zustand 状态管理器保存在前端内存中，随页面刷新而消亡，严禁写入 LocalStorage 以防 XSS 攻击。
  - `refresh_token`：有效期 7 天，**前端安全存储策略**：登录成功时由后端通过 `Set-Cookie` 响应头写入客户端浏览器，必须带有 `httpOnly`、`Secure`、`SameSite=Lax` 属性，前端 JavaScript 脚本无权读取，专门用于在 `access_token` 过期后前端静默自动换取新 Token。
- **“记住密码”的无矛盾底层控制逻辑**：
- 前端在 `/api/v1/auth/login` 请求体中，根据用户是否勾选“记住密码”复选框，显式传递布尔参数 `remember_me`。
- **若用户勾选了“记住密码”（`remember_me: true`）**：后端在下发 `refresh_token` 的 `Set-Cookie` 响应头中，**必须显式指定 `Max-Age=604800`（7天）或 `Expires` 截止时间**。此时浏览器会将其作为 Persistent Cookie（持久化 Cookie）写入磁盘，即使关闭浏览器或电脑重启，该 Cookie 依然存在，下次打开网页时全局 Axios 拦截器捕获到Zustand 内存中无 `access_token`，自动静默请求 `/api/v1/auth/refresh` 接口（浏览器会自动隐式携带该 Cookie），换取新 `access_token`，实现长效免密登录。
- **若用户未勾选“记住密码”（`remember_me: false`）**：后端在下发 `refresh_token` 的 `Set-Cookie` 响应头时，**必须完全忽略、绝不声明 `Max-Age` 和 `Expires` 属性**。此时浏览器会将该 Cookie 强制降级定义为 **Session Cookie（会话级 Cookie）**。一旦用户关闭当前浏览器标签页或关闭浏览器，该 Cookie 将被浏览器底层自动瞬间抹除，生命周期完美终结，彻底消除逻辑冲突。前端不处理、不感知、不存储 `refresh_token`，彻底消除逻辑冲突。
  - **🔄 Token 静默刷新失败降级路径（异常流定义）**：若 Axios 拦截器捕获到 401 错误并调用 `/auth/refresh` 刷新失败（原因包括：refresh_token 已过 7 天有效期、服务端主动吊销、网络故障等），前端必须执行以下降级链：① 调用 `clearAuth()` 清除 Zustand 内存中的 `access_token` 和用户信息；② 清除浏览器中持久化的认证状态；③ 静默重定向至 `/login` 页面。用户感受到的体验为"长时间未操作后需重新登录"，而非技术错误堆栈。
- **Bcrypt 自适应参数验证规范**：
  - 后端在验证用户密码时，必须直接调用安全库自带的 `verify(plain_password, hashed_password)` 函数。该函数会自动解析数据库中存储哈希串头部的算法版本和工作因子，并以该配置执行动态自适应验证。
  - 严禁在校验时死板校验全局默认 rounds 参数，确保数据库中预置的种子哈希不会因后端后续调整全局 rounds 而发生首登登录失败。
- **前端路由规划**：
  - `/login`：登录页。
  - `/register`：注册页。
  - `/dashboard`：普通用户主工作台（包含上传、历史批次列表、多文件对比视图）。
  - `/admin`：管理员全局看板（非管理员访问强制重定向至 `/dashboard`）。
- 权限拦截：
  - 前端 React 建立全局路由守卫，未登录用户重定向至 `/login`；非管理员用户尝试访问 `/admin/`* 路径时强制拦截并重定向至首页。
  - 后端 FastAPI 编写专属依赖项 `verify_admin`，对所有管理端 API 接口进行 Token 角色强校验，非管理员请求直接抛出 `403 Forbidden` 异常。
- **后端拦截机制**：
  - 设计 `verify_token` 依赖项作为全局网关拦截器，解析 JWT 并注入当前上下文，同时强验用户 `is_active` 状态是否为 TRUE，若为 FALSE 则拒绝访问并返回统一错误响应。
  - 设计 `verify_admin` 专有依赖项，拦截所有 `/api/v1/admin/`* 路径下的请求，非管理员用户直接响应 `403 Forbidden`。
- 个人中心：查看历史处理记录、下载导出文件
- **忘记密码全链路验证码安全自愈与单次使用熔断机制（业务硬约束）**：
  1. **发起申请与模糊防御**：用户输入邮箱请求发送验证码。若邮箱在系统内不存在，必须返回与成功完全一致的通用友好提示，但在后台不执行实际发信，刚性防御黑客恶意枚举系统已有用户。
  2. **真实验证码生成与投递**：针对合法用户，系统通过强安全随机数生成器生成 6 位数字验证码，严禁随接口返回前端。系统调用集成的异步邮件组件依赖标准 SMTP 协议投递。
    - *开发环境沙盒降级*：若环境变量开启 `MAIL_SANDBOX_MODE=true`，系统自动跳过真实 SMTP 链路，改将邮件 HTML 写入本地 `/logs/mock_mails/[时间戳]_[邮箱].html`，确保离线演示功能的绝对完整。
  3. **多重连续请求覆盖熔断**：同一邮箱在 10 分钟内连续多次触发发送时，系统在数据库单一事务中，必须将该邮箱旗下所有当前仍处于 `is_used = FALSE` 且未过期的历史旧码强制更新为 `is_used = TRUE`（或物理删除），确保有且仅有最后一次下发的验证码具有激活状态。
  4. **密码刚性复杂度底线**：新密码长度必须在 8-20 位之间，且必须同时包含大写字母、小写字母、数字及特殊符号（如 `@`, `$`, `!`, `%` 等）中的至少三种。前后端执行完全一致的正则表达式强验。
  5. **提交重置与原子熔断（防重放与爆破）**：用户提交重置请求时，系统必须在同一个数据库本地事务中执行串联校验：
    - 强验验证码是否过期（有效期 10 分钟）；
    - 强验 `is_used` 是否为 FALSE；
    - 强验单个验证码的输错尝试次数 `retry_count` 是否 $\ge 5$（若超限则该码立即废弃，报 400 错误）。
    - 校验通过后，原子更新用户密码，并立刻将当前记录的 `is_used` 置为 `TRUE`。此时该记录转化为历史凭据，`retry_count` 维持成功前计数值并冻结（无需归零），用于后续风控安全合规审计。

### 3.2 PDF 批量处理模块

- **明确目标**：实施刚性的文件上传边界校验，构建高效的 Celery 异步处理流式管道，利用 MD5 建立全局去重与结果复用机制。
- **核心业务边界**：
  - **刚性约束限制**：单次批量上传最大文件数 = 10 个。单文件体积必须 $\le$ 20MB。若任意一项超限，前端立即弹出 AntD Notification 拒绝提交并中断请求。
  - **文件去重策略（MD5 强匹配）**：
  - 后端接收到 PDF 文件流后，立即计算文件的 MD5 值并检索 `reports` 表。
    > **⏱️ MD5 去重检测执行时机（架构刚性约束）**：MD5 去重与秒传判定逻辑必须严格在 `POST /api/v1/files/upload` 同步阶段完成，不得推迟至Celery异步管道。这意味着：①相同MD5文件的秒传直接同步写入 `reports` 行并复用 `raw_markdown`；②异步管道仅负责处理新文件的PDF解析与AI提取。此设计确保——同一批次中10个文件若有8个命中历史记录，仅触发2次异步解析任务，最大化异步队列资源利用效率。
  - 若命中相同 MD5 且状态为 `success` 的历史记录：
    1. **拒绝磁盘二次写入**：不再向本地文件系统写入任何新文件。
    2. **路径级级联复用**：新生成的 `reports` 行记录中，`stored_path` 直接复制引用历史记录的已有物理路径。
    3. **数据直接拷贝**：直接复制其 `raw_markdown` 结果并绑定至新批次，实现真正的秒传。
  - **异步流式管道设计**：
    - **Step 1（解析任务）**：Celery 调度 Python 解析库内 PDF 的文本、表格无损提取并清洗为干净的 Markdown，写入 `reports.raw_markdown`。
    - **Step 2（提取任务）**：若 Step 1 成功，立即触发 AI 指标提取逻辑，将 Markdown 喂给大模型（API 模型参数严格指定为 `deepseek-ai/DeepSeek-V3`）。
    - **Step 3（持久化）**：AI 成功返回数据结构后，解析并存入 `extracted_metrics` 表，更新批次状态为 `completed`。
    - **错误重试规范**：若因 API 超时导致解析失败，Celery 自动触发重试机制（最多重试 3 次，每次间隔 10 秒），若 3 次均失败，状态标记为 `failed`，并记录错误日志。
  - **📄 PDF 页数超限处理策略**：`parse_pdf` 硬性截取前 20 页，超出部分直接丢弃不处理。解析日志中必须记录原始总页数与已丢弃页数，方便用户评估结果完整性。此策略在需求层面与 8 项预设指标的提取场景匹配（港交所翌日披露报表通常不超过 5 页）。
  - **📊 批次状态同步规则**：`upload_batches.status` 与子报告状态保持联动——批次内所有报告 `success` → 批次 `completed`；所有报告 `failed` → 批次 `failed`；部分成功部分失败 → 批次 `partial`。DDL 中 `status` 枚举需包含 `partial` 状态。前端对比矩阵仅在批次状态为 `completed` 或 `partial` 时允许查看。

### 3.3 指标提取与对比输出模块

- **明确目标**：基于静态 Pydantic Blueprint 规范，配合特定的 System Prompt 强约束大模型稳定输出 JSON，并在前端实现基于价格偏离度与资金吞吐量的异常高亮对比矩阵。
- **AI 指标提取协议规范**：
  - **系统预置指标模板**：系统初始化时，为管理员账号预置以下 8 项港股常用指标（`is_system=True`），作为新用户首次使用时的默认兜底选项。指标的所有权通过 `is_system` 标识而非 `user_id` 硬编码判断，确保管理员账号迁移时兜底逻辑不受影响。普通用户可在此基础上通过自定义指标管理接口新增、修改或删除自己的指标，完全支持动态扩展。写入数据库 `extracted_metrics` 表时，其对应的系统内部标识（`metric_name`）与前端显示名（`metric_display_name`）映射关系刚性定义如下：

    | #   | JSON Schema 字段名 (亦作 `metric_name`) | 对应的物理表数据类型                 | 前端显示名 (`metric_display_name`) | 业务标准示例        |
    | --- | ---------------------------------- | -------------------------- | ----------------------------- | ------------- |
    | 1   | `company_name`                     | VARCHAR / TEXT             | 公司名称                          | 腾讯控股有限公司      |
    | 2   | `stock_code`                       | VARCHAR                    | 证券代号                          | 00700         |
    | 3   | `submission_date`                  | VARCHAR (数据写入 fiscal_year) | 呈交日期                          | 2026-06-02    |
    | 4   | `repurchase_date`                  | VARCHAR (数据写入 fiscal_year) | 交易日                           | 2026-06-01    |
    | 5   | `shares_repurchased`               | DECIMAL_num / VARCHAR_raw  | 购回股份数目                        | 1250000       |
    | 6   | `highest_price_paid`               | DECIMAL_num / VARCHAR_raw  | 每股最高购回价                       | 382.4000      |
    | 7   | `lowest_price_paid`                | DECIMAL_num / VARCHAR_raw  | 每股最低购回价                       | 376.2000      |
    | 8   | `total_consideration`              | DECIMAL_num / VARCHAR_raw  | 付出价格总额                        | 47350000.0000 |

  - **大模型调用策略**：调用硅基流动 `deepseek-ai/DeepSeek-V3` 接口时，必须在请求头中开启 `json_object` 响应模式。
  - **指标提取 Prompt 结构定义**（针对港交所表格特征进行章节锚定约束）：
    ```markdown
    你是一个精通港股合规公告的数据审计专家。请仔细阅读用户提供的“翌日披露报表” PDF 文本，精准提取指定的 8 项核心指标。

    【特定定位指导（极其重要）】：
    1. 寻找文本顶部的“公司名稱”和“呈交日期”。
    2. 忽略“第一章節”中关于期权、股份变动的干扰数据。
    3. 必须精准定位到“第二章節 購回報告”（或 A. 購回報告）的表格结构中，提取里面的“交易日”、“購回股份數目”、“每股最高購回價”、“每股最低購回價”以及“付出的價格總額”。

    【数据清洗规整三原则（极为重要，违反将导致系统解析崩溃）】：
    1. 日期格式转化：所有提取的日期（submission_date, repurchase_date）必须统一规整为 "YYYY-MM-DD" 格式。如文本中写着 "2026年6月2日" 或 "02/06/2026"，必须在 JSON 中强制输出为 "2026-06-02"。
    2. 数值纯化剔除：提取的所有数值型字段（shares_repurchased, highest_price_paid, lowest_price_paid, total_consideration）必须输出为纯粹的整型（integer）或浮点型（float）。必须在输出前剔除如 "港元"、"元"、"$"、"股"、"%" 等一切非数字文本，且严禁夹带用作千分位符的逗号（例如，"47,350,000" 必须纯化规整为 47350000.00）。
    3. 证券代码消歧：针对证券代号（stock_code），必须清洗为标准的 5 位主板港股代号字符串（不足 5 位的前面强制补 0，如提取到“700”或“0700”必须规范输出为 "00700"）。若文中同时存在人民币柜台代号（如“80700”），必须直接予以忽略，仅锁定并提取港币柜台主板代号。

    必须严格遵循以下 JSON Schema 输出数据，严禁夹带任何 Markdown 标记（如 ```json）或多余的自然语言解释。
    如果文本中未提及某项指标，该指标的对应字段必须返回 null。

    JSON Schema 约束：
    {
      "company_name": string or null,
      "stock_code": string or null,
      "submission_date": string (YYYY-MM-DD) or null,
      "repurchase_date": string (YYYY-MM-DD) or null,
      "shares_repurchased": integer or null,
      "highest_price_paid": float or null,
      "lowest_price_paid": float or null,
      "total_consideration": float or null
    }
    ```
  - **异常数值检测算法逻辑（联动调整）**：
    > **⚠️ 前提条件（与动态指标系统的联动约束）**：异常检测算法并非全局生效。仅在当前批次绑定的指标集中同时包含 `stock_code`、`highest_price_paid`、`lowest_price_paid`、`shares_repurchased`、`total_consideration` 这 5 项港股核心字段时，才启用异常检测逻辑。若批次绑定的是用户自定义指标集（如学术论文场景），后端自动跳过异常检测，前端对比矩阵不渲染任何异常高亮背景，仅展示纯数据。此设计刚性确保系统在多元场景下的通用性，防止算法因依赖字段缺失而崩溃。
    > 由于购回报告无增长率，且系统不维护外部高频历史股票价格库，前端横向对比矩阵的异常检测逻辑为**“当前处理批次内同股异动监测”**：
    > **【小样本边界防护机制（刚性约束）】**：异常检测算法仅在当前对比批次内、属于同一证券代号（`stock_code`）的有效报告总数量 $N \ge 3$ 份时生效。若 $N < 3$（如单文件上传或批次内仅有 2 份报告），算法自动跳过且前端不进行任何异常高亮提示，以绝对防止分母为零引发的系统崩溃（如后端 `ZeroDivisionError` 或前端代码产生 `NaN`）以及小样本对比带来的高频误报。
    > 当满足 $N \ge 3$ 样本边界时，严格执行以下规则：
    1. **价格价差异常**：当前文件的 `highest_price_paid` 或 `lowest_price_paid` 偏离**“当前处理批次（`batch_id`）中相同证券代号（`stock_code`）报告”**中位数的 $\pm 5$ 以上时，触发价格异常高亮（浅橘色背景）。
    2. **单一交易日吞吐量异动**：若单日购回数量 `shares_repurchased` 或付出总额 `total_consideration` 超过同批次其他报告均值的 200% 以上，该单元格自动渲染为浅红色背景，并附加高头寸警告图标（*注：计算其他报告均值时，分母固定为 $N-1$，确保排除自身数据干扰*）。
  - **数据矩阵与导出规范**：前端将 MySQL 中的多报告数据拼装成双轴矩阵（行标题为当前批次绑定指标集的 `metric_display_name`，列标题为文件名/交易日）。一键导出功能使用前端 `xlsx` 库，直接抓取当前 DOM 表格的二维数据流并本地转换为标准的 `.xlsx` 文件下载。在动态指标场景下，矩阵列数随用户勾选的指标数动态变化，导出表头随之联动。
  - **📐 `fiscal_year` 字段语义规范（跨场景适配约束）**：`fiscal_year` 字段定义为**灵活的时期锚点（Period Anchor）**，而非严格的财务会计术语。根据业务场景不同，该字段可承载：① 具体日期（如 `2026-06-02` 表示购回交易日）；② 财年标识（如 `2025` 表示 FY2025）；③ 学年标识（如 `2024-2025`）。此灵活性是 EAV 纵表架构的刻意设计——不同领域（金融、学术、法律）的报告提取结果可在同一物理表中共存，无需为每种场景单独建表。前端在可视化时按 `reports.created_at` 作为时间轴排序基准，`fiscal_year` 仅作为数据点标签展示。

### 3.4 AI 聊天问答小组件

- **明确目标**：在不破坏页面布局的情况下，为用户提供基于当前单份报告或当前批次报告内容的实时流式追问能力。
- **具体功能与 UI 规范**：
  - **UI 布局表现（绝对约束）**：组件固定悬浮于系统右下角。面板采用折叠式设计，点击悬浮图标展开。**不支持自由拖拽**以简化前端图层冲突。消息气泡样式参考标准 AntD Chat 规范（用户消息为深蓝色靠右，AI 消息为浅灰色靠左，字体统一使用等宽字体）。
  - **双模态上下文注入机制**：
    1. **报告绑定模式**：当用户在前端激活并查看某份报告时，小组件自动将该报告的 `report_id` 注入当前会话。后端直接读取该报告对应的 `raw_markdown` 全文作为 Context，拼接在 System Prompt 之后输入给大模型。
    2. **通用知识模式（无报告绑定）**：当用户在首次打开应用或未选择任何报告的情景下唤醒聊天窗时，`report_id` 传递为 `null`。此时系统自动降级为“通用平台 AI 助手”，System Prompt 切换为基础角色设定，完全基于 `deepseek-ai/DeepSeek-V3` 的自身基础知识储备回答用户的任意通用提问。
  - **流式传输规范**：后端统一采用标准的 **SSE (Server-Sent Events)** 协议进行下行流式传输，以 `text/event-stream` 格式将文本逐字推送到 React 前端聊天窗口，渲染出高效的打字机特效。
  - **对话流控制（窗口滑动机制）**：后端每次只截取并携带当前 `session_id` 下最新的 5 轮（10 条历史记录）对话记录随同文档内容一起发送，超出 5 轮的历史数据自动从当前请求大模型的 Prompt 中滑出，但依然保留在数据库中。
  - **安全限流策略 (Rate Limiting)**：针对单个用户标识，聊天接口在后端部署令牌桶算法限流规则：每分钟最大允许发送 10 条消息，超过限制后端返回 `429 Too Many Requests` 标准响应。前端同时做客户端友好提示，但核心限流逻辑必须以后端为准（前端限流可被 DevTools 绕过）。

### 3.5 后台管理

- **明确目标**：为管理员提供全局数据看板、用户活动审计和处理日志监控能力。
- **具体功能**：

- 用户看板：以 AntD Table 展示系统内所有已注册用户的列表（包含注册时间、累计上传文件数）。
- **全局用户管理**：通过 AntD Table 罗列系统内所有用户的注册时间、角色属性与累计处理批次数量，支持禁用/启用普通用户状态。
- **穿透式审计管理**：管理员可以调取系统内任意用户创建的 `upload_batches` 和 `reports` 记录。点击”查看详情”可直接调阅该报告在数据库中存储的 `raw_markdown` 原文以及大模型输出的原始 `JSON` 结构体，用于系统调优。

#### 3.5.1 个人中心与头像管理

- **明确目标**：为用户提供个人信息的自助管理入口，包括头像上传和基本资料修改。
- **具体功能**：
  - **头像上传**：用户可在个人中心上传/更换头像。支持 JPG/PNG 格式，单文件限制 2MB。头像存储于 `uploads/avatars/` 目录，通过静态文件服务对外提供访问。上传新头像时自动删除旧头像文件。
  - **个人信息编辑**：用户可修改昵称（直接生效）；修改邮箱时需提供当前密码进行身份验证，且新邮箱不可被其他用户占用。
  - **User 模型扩展**：`users` 表新增 `avatar_url` 字段（`VARCHAR(500), NULLABLE`），存储头像的 URL 路径。
- **业务规则**：
  - API 路由前缀：`/api/v1/auth/me`
  - `PUT /auth/me` — 更新昵称/邮箱（需当前密码）
  - `POST /auth/me/avatar` — 上传头像（multipart，返回 `{ avatar_url }`）
- **前端路由**：`/profile` — 个人中心页面（需登录），通过右上角用户菜单「个人中心」入口进入。

#### 3.5.2 批次与报告删除

- **明确目标**：允许用户清理不再需要的上传批次和报告，释放存储空间。
- **具体功能**：
  - **删除批次**：用户可删除自己创建的整个批次。删除操作级联清除所有关联报告、指标数据，并从磁盘物理删除 PDF 文件。管理员可删除任意用户的批次。
  - **删除单个报告**：用户可删除批次内的单个报告（含关联指标和磁盘文件）。若批次内仅剩此报告，则同时删除整个批次。
  - **操作确认**：前端通过 `Popconfirm` 二次确认后才执行删除，防止误操作。
- **业务规则**：
  - 采用**物理删除**策略（数据库记录 + 磁盘文件均永久删除）
  - API：`DELETE /api/v1/batches/{batch_id}` — 删除批次
  - API：`DELETE /api/v1/files/reports/{report_id}` — 删除单个报告
  - 权限：普通用户仅可操作自己的文件，管理员可操作任意用户的文件
  - 磁盘文件删除失败不影响数据库事务（已提交后异步清理）

#### 3.5.3 批次与报告重命名

- **明确目标**：支持用户在上传时自定义批次名称，以及上传后对批次和报告进行重命名。
- **具体功能**：
  - **上传时命名**：上传区域新增「批次名称」输入框（可选），用户可输入自定义名称。留空则沿用默认命名规则。
  - **批次重命名**：批次列表中双击批次名称可触发内联编辑，按回车或失焦时自动保存。
  - **报告重命名**：提供 API 端点修改报告的 `original_filename` 字段。
- **业务规则**：
  - 名称长度限制：1-255 字符，不可为空
  - API：`PUT /api/v1/batches/{batch_id}` — 重命名批次（`{ “batch_name”: “...” }`）
  - API：`PUT /api/v1/files/reports/{report_id}` — 重命名报告（`{ “original_filename”: “...” }`）
  - 权限：普通用户仅可操作自己的文件，管理员可操作任意用户的文件

### 3.6 系统接口 (API) 契约规范

系统内部所有通信接口必须严格遵循以下 RESTful 端点设计（全局前缀：`/api/v1`）：
无论是后端校验拦截、业务报错还是未捕获的系统异常，接口返回给前端的非 2xx 状态码错误响应结构必须严格统一为以下 JSON Schema：

```json
{
  "code": 400,                          -- 具体的 HTTP 状态码或业务错误码
  "message": "用户已被禁用/邮箱已存在", -- 面向用户的友好提示文本
  "details": null                       -- 详细错误调试堆栈或表单校验错误列表，无详情时为 null
}
```

#### 3.6.1 认证分类接口 (Auth)

1. **POST `/api/v1/auth/register`** - 描述：用户注册
  > **🔐 密码复杂度刚性约束**：注册密码复杂度要求与忘记密码重置保持一致——长度 8-20 位，必须同时包含大写字母、小写字母、数字及特殊符号（如 `@`、`$`、`!`、`%` 等）中的至少三种。前端和后端必须执行完全一致的正则表达式校验。若密码不合规，后端返回 `422 Unprocessable Entity`。
  - 请求载荷：`{ "email": "string", "password": "string", "nickname": "string" }`  
  - 响应码：`201 Created`（成功），`400 Bad Request`（邮箱已存在/格式错误）
2. **POST `/api/v1/auth/login`** - 描述：用户登录认证
  - 请求载荷：`{ "email": "string", "password": "string" }`  
  - 响应内容：`{ "access_token": "string", "refresh_token": "string", "is_admin": boolean }`  
  - 响应码：`200 OK`，`401 Unauthorized`（密码错误/账号被禁用）
3. **POST `/api/v1/auth/refresh`** - 描述：静默刷新 Access Token
  - 请求头：携带有效的 `Refresh-Token`  
  - 响应内容：`{ "access_token": "string" }`
4. **POST `/api/v1/auth/forgot-password/send-code`** - 描述：申请忘记密码验证码
  - **请求载荷 (JSON Body)**：`{ "email": "user@example.com" }`
  - **后端核心逻辑**：执行频率限制（60秒/次），调用异步邮件组件发送验证码，**具体安全模糊防御与沙盒降级逻辑详见 3.1 节业务约束**。
  - **成功响应 (JSON / 200 OK)**：
    ```json
    {
      "status": "success",
      "message": "验证码已成功发送至您的注册邮箱，请在 10 分钟内完成验证修改。"
    }
    ```
  - **异常响应 (JSON / 503 Service Unavailable)**：
    ```json
    { "code": 503, "message": "邮件投递服务暂时不可用，请联系管理员检查 SMTP 配置。" }
    ```
5. **POST `/api/v1/auth/forgot-password/reset-with-code`** - 描述：提交验证码并执行密码重置
  - **鉴权**：无需 Access Token（匿名端点）
  - **请求载荷 (JSON Body)**：
    ```json
    {
      "email": "user@example.com",
      "verification_code": "482015",
      "new_password": "NewSecurePassword@123"
    }
    ```
  - **后端核心逻辑**：**系统必须严格在单一数据库本地事务（Database Transaction）中串联执行 3.1 节所述的“提交重置与原子熔断机制”（包含过期、超限、撞码校验及密码复杂度强验）。此处不再冗述。**
  - **成功响应 (JSON / 200 OK)**：
    ```json
    {
      "status": "success",
      "message": "密码修改成功，原验证码已失效销毁。请使用新密码重新登录。"
    }
    ```
  - **异常响应①（密码复杂度未达标 / 422 Unprocessable Entity）**：
    ```json
    { "code": 422, "message": "密码复杂度未达标。新密码长度需在8-20位之间，且必须同时包含大写字母、小写字母、数字及特殊符号中的至少三种。" }
    ```
  - **异常响应②（验证码错误或输错达5次锁死 / 400 Bad Request）**：
    ```json
    { "code": 400, "message": "验证码无效、已过期或已被重复使用，请重新获取。" }
    ```
  - **常见响应码**：`200 OK`（重置成功），`400 Bad Request`（验证码失效/过期/重试超限），`422 Unprocessable Entity`（密码不合规）
6. **PUT `/api/v1/auth/me`** — 描述：更新当前用户的个人信息（昵称、邮箱）
  - **鉴权**：需要有效 Access Token
  - 请求载荷：`{ "nickname"?: string, "email"?: string, "current_password"?: string }`
  - 业务规则：修改邮箱时需提供 `current_password` 验证；新邮箱不可被其他用户占用
  - 响应内容：更新后的 `UserResponse`
7. **POST `/api/v1/auth/me/avatar`** — 描述：上传用户头像
  - **鉴权**：需要有效 Access Token
  - **请求类型**：`multipart/form-data`，字段名 `file`
  - 限制：仅 JPG/PNG，最大 2MB
  - 响应内容：`{ "avatar_url": "/uploads/avatars/{user_id}_{uuid}.jpg" }`
  - 业务规则：上传新头像时自动删除旧头像文件
8. **GET `/api/v1/auth/me`** — 描述：获取当前登录用户的完整信息（已有接口，响应增加 `avatar_url` 字段）

#### 3.6.2 文件与批次处理接口 (Files & Batches)

1. **POST `/api/v1/files/upload`** - 描述：批量上传 PDF 并创建解析批次、创建解析批次并绑定选定的提取指标
  - **鉴权**：需要有效的 Access Token（Header: `Authorization: Bearer <token>`）
  - **请求类型**：`multipart/form-data`
  - **请求载荷 (Form Data)**：
    - `files`: `File[]` (二进制文件列表，严格限制最多 10 个且单个 $\le$ 20MB)
    - `batch_name`: `string` (用户自定义的批次名称)
    - `metric_ids`: `int[]` (用户本次勾选的指标 ID 列表。前端必须以多个同名 FormData 字段 `metric_ids=1&metric_ids=3&metric_ids=5` 的方式传递；后端 FastAPI 统一使用 `List[int] = Form(...)` 接收。**已废弃逗号分隔字符串方案，避免数字字符串解析歧义。**)
  - **后端核心逻辑**：
  1. 验证文件数量与大小，计算每个文件的 MD5 值进行全局去重/秒传判定。
  2. 在 `upload_batches` 表中创建一条初始状态为 `'pending'` 的批次记录，获取生成自增的 `batch_id`。
  3. 解析 `metric_ids` 参数，遍历用户勾选的指标 ID，顺次向 `batch_metric_relations` 表中插入多条关联映射，锁定当前批次与指标集的绑定关系。
  4. 关联成功后，正式唤醒 Celery 异步流式管道（将 `batch_id` 投递给异步队列）。
    成功响应内容 (JSON / 202 Accepted)**：
    `json
    "status": "success",
    "batch_id": 42,
    "message": "文件上传及动态指标集绑定成功，后台 Celery 异步解析任务已平滑调度接收。"
     `常见响应码**：`202 Accepted`（任务接收并开始异步处理），`400 Bad Request`（文件超限/缺少必要的指标勾选），`401 Unauthorized`（Token 失效）
2. **GET `/api/v1/batches`** - 描述：获取当前用户的所有历史处理批次列表（严格执行分页契约）
  - **查询参数（Query）**：`page`（必须，整数，默认 1），`page_size`（必须，整数，默认 10）  
  - **成功响应内容 Schema**：
    ```json
    {
      "total": 125,                    // 系统内符合当前查询条件的总记录数
      "page": 1,                       // 当前返回的页码
      "page_size": 10,                 // 当前单页限制条数
      "items": [                       // 数据包裹数组
        {
          "batch_id": 1,
          "batch_name": "2026年Q2财报比对批次",
          "status": "completed",
          "created_at": "2026-06-05 22:33:42"
        }
      ]
    }
    ```
3. **GET `/api/v1/batches/{batch_id}`** - 描述：获取特定批次的详细多报告对比矩阵数据
  - 响应内容：返回该批次下所有报告的状态、文件名以及提取出的标准化指标 JSON 聚合体。
4. **DELETE `/api/v1/batches/{batch_id}`** — 描述：删除批次及其所有关联报告、指标和 PDF 磁盘文件
  - **鉴权**：需要有效 Access Token；用户仅可删除自己的批次，管理员可删除任意用户
  - 响应内容：`{ "message": "批次 #N 已删除" }`
5. **PUT `/api/v1/batches/{batch_id}`** — 描述：重命名批次
  - **鉴权**：需要有效 Access Token；用户仅可操作自己的批次，管理员可操作任意用户
    - 请求载荷：`{ "batch_name": "string" }` (1-255 字符)
    - 响应内容：`{ "message": "批次已重命名", "batch_name": "..." }`
6. **DELETE `/api/v1/files/reports/{report_id}`** — 描述：删除单个报告（含关联指标和磁盘文件）
  - **鉴权**：需要有效 Access Token；用户仅可删除自己的报告，管理员可删除任意用户
    - 业务规则：若批次内仅剩此报告，则同时删除整个批次；否则仅删除此报告并更新批次计数
    - 响应内容：`{ "message": "报告 #N 已删除" }` 或 `{ "message": "报告 #N 及所属批次已删除" }`

### 3.7 动态指标扩展与多维可视化分析规范

#### 3.7.1 业务流程与架构解耦

- **指标动态配置化**：系统彻底废除对大模型提取指标名称（如 `revenue`, `buyback_shares`）的后端硬编码。大模型“提取什么”完全取决于用户前台的动态配置与勾选，从而使系统具备处理金融财报、学术论文、法律合同等多元化文件的通用能力。
- **数据存储通用化（纵表映射）**：系统充分利用 `extracted_metrics` 表的纵表（EAV 实体-属性-值）结构特性。当用户自定义新增指标时，后端仅在配置表中增加一条定义元数据，在提取结果表中增加行记录，**严禁在物理表结构上执行 `ALTER TABLE` 操作**。

#### 3.7.2 模型稳定输出与后端对齐存储规范

- **默认兜底与自定义扩展双模混合分流机制（核心路由控制）**：
为了同时兼顾“开箱即用”的高效用户体验与“多元场景”的动态灵活性，系统必须支持双模混合解析逻辑，在后端路由层实现静默无感分流：
  1. **默认模式**：当用户未在前端配置或勾选任何自定义指标，直接提交文件上传请求时，POST `/api/v1/files/upload` 接口接收到的 `metric_ids` 字段为空。此时，后端 FastAPI 必须执行兜底逻辑，自动读取数据库中由系统管理员初始化注入的 8 项港股预置核心指标（即 `user_id = 1` 的默认种子数据），并在事务中将这 8 个默认指标 ID 与当前批次绑定写入 `batch_metric_relations` 表。
    > **🔑 `is_system` 标识机制（防歧义澄清）**：系统预置指标通过 `is_system=True` 标识实现全局可见，对系统内所有用户透明开放选择。`user_id=1` 仅标记该预置指标的创建者身份，系统在判断默认兜底指标集时，必须使用 `is_system=True` 条件查询，严禁依赖 `user_id=1` 硬编码判断。当管理员账号发生迁移或ID变更时，预置指标集不应受影响。
  2. **自定义动态模式（按需扩展）**：当用户显式勾选了特定的指标集时，接口接收到具体的 `metric_ids`（如 `"1,3,5"`），后端则直接按照用户指定的指标集进行物理绑定。
  3. **架构统一约束**：经后端分流补齐后，Celery 异步任务与大模型动态 Prompt 组装层**不感知双模差异，统一视作动态批次处理**。这刚性确保了系统核心逻辑的绝对高内聚，既实现了港股报表的“零配置秒级一键提取”，又保留了全行业文件兼容的扩展潜力。
- **API 刚性参数控制**：在 FastAPI 后端调用硅基流动平台的 DeepSeek-V3 API 时，必须强制传入参数 `response_format={"type": "json_object"}`，从底层确保大模型返回的数据必然为标准、可解析的 JSON 字典，杜绝 Markdown 代码块杂质。
- **运行期动态指标集过滤机制（解决指标集确定问题）**：
  - 用户在前台上传文件时，系统必须提供指标勾选矩阵（支持一键全选或部分勾选）。
  - 前端提交上传请求时，后端在生成 `upload_batches` 记录的同时，必须将用户本次勾选的指标 ID 顺次写入 `batch_metric_relations` 关联表中。
  - Celery 异步任务在启动时，**必须根据当前 `batch_id` 联查 `batch_metric_relations` 表**，仅获取当前批次绑定的指标子集。严禁盲目读取用户全量指标，确保 Prompt 指令的高聚焦度与 Token 经济性。
- **expected_type 与底层存储字段对齐清洗策略（解决数据分流与防崩溃问题）**：
后端 Celery 任务在解析大模型返回的动态 JSON 字典时，必须读取该指标在 `metric_definitions` 中声明的 `expected_type`，并严格执行以下数据路由与清洗规则：
  1. **当 `expected_type == 'NUMERIC'`（数值型指标）时**：
    - 后端必须对大模型返回的原始字符串进行**强类型正则清洗**。抽取其中的纯数字与小数点，剔除“元”、“%”、“倍”等单位干扰。
    - 若成功清洗出标准浮点数，将其转换为 `Float/Decimal` 类型，写入 `extracted_metrics` 表的 `metric_value_num` 字段；同时将未清洗的完整原始文本写入 `metric_value_raw` 中备份留痕。
    - 若大模型返回非数字文本（如“未能提及”、“不适用”、“null”），系统必须将 `metric_value_num` 置为 `NULL`，将该异常原始文本写入 `metric_value_raw`，**绝对禁止将脏字符串强行写入数值字段以防数据库崩溃**。
  2. **当 `expected_type == 'TEXT'`（文本型指标）时**：
    - 后端直接跳过数值强转链路。
    - 将大模型返回的文本内容直接完整写入 `extracted_metrics` 表的 `metric_value_raw` 字段，同时将对应的 `metric_value_num` 刚性置为 `NULL`。
- **工程解析防错兜底**：若大模型在某次响应中偶发性缺失了某项被绑定的 Key，后端必须捕获该异常，自动为该指标生成一条 `metric_value_num = NULL` 且 `metric_value_raw = "AI未输出该指标内容"` 的兜底容错记录，确保整批文件解析任务的刚性健壮性。

#### 3.7.3 多形态数据可视化对比规范

- **技术选型**：前端在 React 18 + Ant Design 技术栈基础上，轻量级引入官方无缝支持的图表库 `@ant-design/charts`（基于 G2 引擎）。
- **多维图形分析场景**：
  1. **单指标时序趋势分析（折线图/面积图）**：
    - **触发条件**：当用户查看同一实体（如某特定公司、特定研究对象）跨越多个不同 `fiscal_year`（时期/财年）的数据时触发。
    - **数据转换**：前端提取该指标对应的 `metric_value_num` 作为 Y 轴，`fiscal_year` 排序后作为 X 轴，自动渲染趋势折线，使用户一目了然数据的演变方向。
  2. **多主体横向跨文件对比（柱状图/条形图）**：
    - **触发条件**：当用户在数据表格中勾选同一个 `upload_batches`（上传批次）内的多份不同报告（如对比 3 家竞争对手的同一年份财报）时触发。
    - **数据转换**：前端图表组件以 `report_id`（映射为文件名称）作为分类轴，将各报告中相同的 `metric_name` 的数值作为柱状高度进行并列渲染，方便用户快速进行横向数据大比拼。

## 4. 数据库设计（MySQL 8.0.30 物理表结构）

### 4.1 初始化规范与种子数据硬性约束

- **初始化规范要求**：请严格按照以下 DDL 建表脚本初始化数据库。
- **默认管理员种子数据**：下方 DDL 脚本末尾已包含一条**默认管理员账号的初始化插入语句**。
  - **登录账号（Email）**：`admin@deepreader.com`
  - **初始明文密码**：`Admin@123456`
- **后端安全防篡改校验（研发刚性要求）**：后端鉴权与数据库初始化模块在项目启动时，必须前置执行防御性校验。若检测到此种子用户的哈希被意外篡改、丢失或比对失败，后端代码必须具备自动重置、重新覆盖插入下方合法的 Bcrypt 哈希字符串的逻辑，以绝对确保系统首个全权管理员通道畅通。

### 4.2 物理表 DDL 脚本

```sql
CREATE DATABASE IF NOT EXISTS deepreader_db CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
USE deepreader_db;
-- 1. 用户表
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,
    email VARCHAR(255) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    nickname VARCHAR(100) NOT NULL,
    avatar_url VARCHAR(500) NULL,        -- 用户头像 URL 路径（例如 /uploads/avatars/1_abc123.jpg）
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NULL ON UPDATE CURRENT_TIMESTAMP,
    is_admin BOOLEAN DEFAULT FALSE, -- TRUE 代表管理员，FALSE 代表普通用户
    is_active BOOLEAN DEFAULT TRUE  -- TRUE 代表启用，FALSE 代表禁用（配合管理端操作）
) ENGINE=InnoDB;

-- 2 忘记密码验证码安全审计表（重构：完美支持新码熔断旧码、重试上限与风控留痕）
CREATE TABLE password_reset_codes (
    id INT AUTO_INCREMENT PRIMARY KEY,
    email VARCHAR(255) NOT NULL,
    code_hash VARCHAR(64) NOT NULL,            -- 存储6位验证码的 SHA-256 哈希值，严禁明文存储
    retry_count INT DEFAULT 0,                 -- 累计输错尝试次数，单码达 5 次自动报废
    is_used BOOLEAN DEFAULT FALSE,             -- 状态机核心：重置成功时、或被新请求的验证码覆盖废弃时，统一置为 TRUE
    expired_at TIMESTAMP NOT NULL,             -- 刚性生命周期控制（通常为创建时间 + 10分钟）
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_email_lookup (email, is_used, expired_at) -- 高频检索与覆盖状态更新时的复合索引
) ENGINE=InnoDB COMMENT='忘记密码验证码生命周期与风控审计表';


-- 3. 用户自定义指标配置表（新增：支持租户级动态指标解耦与 AI 提示词引导）
CREATE TABLE metric_definitions (
    id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT NOT NULL,                      -- 归属于具体用户，实现租户级指标隔离
    metric_key VARCHAR(100) NOT NULL,          -- 传给大模型的标准 JSON Key，如: net_profit, sample_size
    metric_label VARCHAR(100) NOT NULL,        -- 前端界面显示的中文名，如: 净利润, 样本量
    expected_type ENUM('NUMERIC', 'TEXT') DEFAULT 'NUMERIC', -- 期望类型，便于后端分流写入 num 或 raw 字段
    prompt_instruction VARCHAR(500) NULL,      -- 指导 AI 提取该指标时的专属微型提示词引导
    is_system BOOLEAN DEFAULT FALSE,             -- TRUE 表示系统预置指标（全局可见），FALSE 表示用户自定义；系统判断默认指标时以此字段为准，严禁依赖 user_id 硬编码
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NULL ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uidx_user_key (user_id, metric_key),
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
) ENGINE=InnoDB COMMENT='用户自定义指标配置表';

-- 4. 上传批次表（一次上传动作）
CREATE TABLE upload_batches (
    id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT NOT NULL,
    batch_name VARCHAR(255) NOT NULL,
    status ENUM('pending', 'processing', 'completed', 'failed', 'partial') DEFAULT 'pending',
    total_files INT DEFAULT 0,                   -- 本批次包含的文件总数
    processed_files INT DEFAULT 0,               -- 已完成处理的文件数（含成功与失败）
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NULL ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    INDEX idx_user_batches (user_id)
) ENGINE=InnoDB;

-- 5. 批次与自定义指标关联表（新增：解决批次与指标集的绑定断层）
CREATE TABLE batch_metric_relations (
    id INT AUTO_INCREMENT PRIMARY KEY,
    batch_id INT NOT NULL,                     -- 关联的上传批次 ID
    metric_def_id INT NOT NULL,                -- 关联的用户自定义指标定义 ID
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uidx_batch_metric (batch_id, metric_def_id),
    FOREIGN KEY (batch_id) REFERENCES upload_batches(id) ON DELETE CASCADE,
    FOREIGN KEY (metric_def_id) REFERENCES metric_definitions(id) ON DELETE CASCADE
) ENGINE=InnoDB COMMENT='上传批次与自定义指标集多对多关联映射表';

-- 6. 单个报告表
CREATE TABLE reports (
    id INT AUTO_INCREMENT PRIMARY KEY,
    batch_id INT NOT NULL,
    original_filename VARCHAR(255) NOT NULL,
    stored_path VARCHAR(500) NOT NULL,
    pdf_md5 VARCHAR(32) NOT NULL,
    file_size INT NOT NULL,                    -- 文件大小（字节），用于审计追踪
    raw_markdown LONGTEXT NULL,                -- 20页内的全量文本直接存储；超限页丢弃并在日志记录
    entity_name VARCHAR(255) NULL,             -- 实体名称（由AI提取的company_name回填），用于跨批次趋势查询的性能优化冗余字段
    parse_status ENUM('pending', 'parsing', 'extracting', 'success', 'failed') DEFAULT 'pending',
    error_message TEXT NULL,                   -- 解析或提取失败时的错误详情，用于调试和用户反馈
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NULL ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (batch_id) REFERENCES upload_batches(id) ON DELETE CASCADE,
    INDEX idx_pdf_md5 (pdf_md5)
) ENGINE=InnoDB;

-- 7. 提取结果表（多维标准化设计，完美支持前端横向对比矩阵）
CREATE TABLE extracted_metrics (
    id INT AUTO_INCREMENT PRIMARY KEY,
    report_id INT NOT NULL,
    metric_name VARCHAR(100) NOT NULL,       -- 系统内部标识，如: revenue, buyback_shares
    metric_display_name VARCHAR(100) NOT NULL, -- 前端显示名，如: 营业收入, 购回股份数量
    metric_value_num DECIMAL(18, 4) NULL,     -- 强类型数值，方便前端排序、过滤和做异常检测
    metric_value_raw VARCHAR(500) NULL,       -- 修复：允许为 NULL。当 AI 未能提取到任何相关文本时填入 NULL
    fiscal_year VARCHAR(20) NULL,             -- 灵活的时期锚点（Period Anchor），可承载日期（如2026-06-02）、财年（如2025）、学年（如2024-2025）；前端按 report.created_at 排序，此字段仅作标签展示
    unit VARCHAR(50) NULL,                    -- 单位，如: "元", "股", "%"
    confidence FLOAT DEFAULT 1.0,             -- AI 置信度
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NULL ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (report_id) REFERENCES reports(id) ON DELETE CASCADE,
    INDEX idx_report_metric (report_id)
) ENGINE=InnoDB;

-- 8. 聊天会话表（会话级元数据管理，与会话内容解耦）
CREATE TABLE chat_sessions (
    id INT AUTO_INCREMENT PRIMARY KEY,
    session_id VARCHAR(100) NOT NULL UNIQUE,    -- 前端生成的会话标识，格式: s_{timestamp}_{random}
    user_id INT NOT NULL,                       -- 会话所属用户
    report_id INT NULL,                         -- 绑定的报告 ID，NULL 表示通用知识模式
    is_active BOOLEAN DEFAULT TRUE,             -- 会话是否活跃
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NULL ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY (report_id) REFERENCES reports(id) ON DELETE SET NULL,
    INDEX idx_session_lookup (session_id),
    INDEX idx_user_lookup (user_id)
) ENGINE=InnoDB COMMENT='聊天会话元数据表';

-- 9. 聊天消息表（消息内容存储，联表查询 chat_sessions）
CREATE TABLE chat_messages (
    id INT AUTO_INCREMENT PRIMARY KEY,
    session_id INT NOT NULL,                    -- 关联 chat_sessions.id（数据库内部 ID）
    role VARCHAR(20) NOT NULL,                  -- 'user' 或 'assistant'
    content TEXT NOT NULL,                      -- 消息正文
    model_used VARCHAR(50) DEFAULT 'deepseek-ai/DeepSeek-V3',  -- 生成此消息的模型
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (session_id) REFERENCES chat_sessions(id) ON DELETE CASCADE,
    INDEX idx_session_history (session_id)       -- 5轮滑动窗口高频加载索引
) ENGINE=InnoDB COMMENT='聊天消息内容表';

## 6. 日志与监控策略

### 6.1 日志规范

| 层级 | 日志工具 | 输出目标 | 级别 |
|------|----------|----------|------|
| FastAPI 请求日志 | Python `logging` 模块 | `stdout` + `backend/logs/app.log` | `INFO`（生产）/ `DEBUG`（开发） |
| Celery 任务日志 | Celery 内置日志 | `stdout`（worker 进程输出） | `INFO` |
| PDF 解析错误 | `print()` → 后续迁移至 `logging` | `stdout` | `ERROR` |
| AI API 调用错误 | `print()` → 后续迁移至 `logging` | `stdout` | `ERROR` |
| 邮件发送日志 | Python `logging` | `stdout` + `backend/logs/email.log` | `INFO` |
| 安全敏感操作 | Python `logging`（`auth` 模块） | `backend/logs/security.log` | `WARNING`（登录失败、验证码爆破、权限越权等） |

### 6.2 Celery 任务监控

- **本地开发**：通过 Celery 命令行 `--loglevel=info` 查看 worker 实时日志。
- **生产环境**：推荐部署 **Celery Flower**（`pip install flower`）作为 Web 监控面板，默认端口 5555，可查看任务队列长度、成功率、失败堆栈、worker 状态等。
  ```bash
  celery -A app.tasks.celery_app flower --port=5555
```

### 6.3 关键告警指标


| 指标           | 告警阈值          | 处理建议                            |
| ------------ | ------------- | ------------------------------- |
| Celery 任务失败率 | > 10% 持续 5 分钟 | 检查 SiliconFlow API 可用性及配额       |
| PDF 解析成功率    | < 90%         | 检查 PyMuPDF 兼容性及 PDF 文件损坏率       |
| 邮件发送失败率      | > 0%（生产环境）    | 检查 SMTP 配置及网络连通性                |
| Token 刷新失败率  | 突增            | 检查 JWT SECRET_KEY 一致性及 Redis 连接 |


### 6.4 审计日志保留

- 安全审计日志（登录/IP/验证码操作）保留 **90 天**。
- 应用日志保留 **30 天**，超出自动轮转。
- 日志格式统一为：`[时间戳] [级别] [模块] [用户ID] 消息内容`。
