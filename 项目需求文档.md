# DeepReader Agent 项目开发说明书

## 1. 项目概述

**项目名称**：DeepReader Agent  
**目标**：开发一个 Web 应用，支持用户登录/注册，能够批量上传 PDF 研报/论文，利用国产大模型自动精准提取核心指标（如营收、增长率、样本量等），并输出对比表格（Excel 或 Markdown）。同时集成基于硅基流动平台的 AI 聊天问答系统，提供基于当前报告内容的辅助解读能力。系统还提供多维数据可视化分析（折线图、柱状图、饼图、雷达图、仪表盘、热力图）和 AI 智能指标推荐功能。  
**适用场景**：金融研报分析、学术论文批量对比、数据调研团队。  
**技术约束**：前端采用 React 18 + TypeScript + Vite，后端采用 Python FastAPI，数据库使用 MySQL 8.0.30，大模型通过硅基流动平台统一调用 DeepSeek-V3，任务异步处理采用 Celery + Redis。

## 3. 功能模块、业务边界与设计细节

### 3.1 用户与权限管理模块

- **明确目标**：建立基于角色访问控制（RBAC）的身份隔离体系，支持 Token 双重刷新机制。普通用户仅能查看和对比自己上传的数据；管理员拥有全局最高权限，可穿透查看所有用户的数据。
- **具体功能规范**：

- 注册：两步验证码流程。① 输入邮箱 → 系统发送 6 位数字验证码（60 秒频率限制）；② 输入验证码 + 密码 + 昵称 → 创建账号。系统对邮箱格式进行严格正则校验（含域名 TLD 合法性判断），密码在后端采用 `passlib (bcrypt)` 进行单向哈希加密存储。新注册账号默认处于激活状态。*管理员账号不提供开放注册通道，仅能通过数据库初始脚本注入。*
- **登录与鉴权（双 Token 策略）**：
  - `access_token`：有效期 15 分钟，载荷中包含 `user_id`、`email`、`is_admin`。**前端安全存储策略**：由 Zustand 状态管理器保存在前端内存中，随页面刷新而消亡，严禁写入 LocalStorage 以防 XSS 攻击。
  - `refresh_token`：有效期 7 天，**前端安全存储策略**：登录成功时由后端通过 `Set-Cookie` 响应头写入客户端浏览器，必须带有 `httpOnly`、`Secure`、`SameSite=Lax` 属性，前端 JavaScript 脚本无权读取，专门用于在 `access_token` 过期后前端静默自动换取新 Token。
- **"记住密码"控制逻辑**：
  - 前端在 `/api/v1/auth/login` 请求体中，根据用户是否勾选"记住密码"复选框，显式传递布尔参数 `remember_me`。
  - 若用户勾选`remember_me: true`：后端在 `Set-Cookie` 响应头中显式指定 `Max-Age=604800`，浏览器将其作为持久化 Cookie 写入磁盘，实现长效免密登录。
  - 若用户未勾选`remember_me: false`：后端不声明 `Max-Age` 和 `Expires`，浏览器将其作为会话级 Cookie，关闭浏览器后自动抹除。
  - **Token 静默刷新失败降级路径**：若 Axios 拦截器捕获到 401 错误并调用 `/auth/refresh` 刷新失败，前端执行降级链：① 调用 `clearAuth()` 清除内存 Token 和用户信息；② 调用 `chatStore.resetAll()` 完全清除聊天数据；③ 静默重定向至 `/login` 页面。
- **Bcrypt 自适应参数验证规范**：后端在验证用户密码时，直接调用 `verify(plain_password, hashed_password)` 函数，自动解析数据库哈希串头部的算法版本和工作因子。严禁在校验时死板校验全局默认 rounds 参数，确保预置种子哈希不受后续全局 rounds 调整的影响。
- **前端路由规划**：
  - `/login`：登录/注册双 Tab 页
  - `/forgot-password`：忘记密码三步骤页
  - `/dashboard`：主工作台（上传、批次列表、对比矩阵、图表分析）
  - `/analytics`：数据分析中心（多批次多指标可视化分析）
  - `/metrics`：指标库（自定义指标管理、AI 推荐、模板导入）
  - `/chat`：AI 对话页（DeepSeek 风格全屏聊天界面）
  - `/files`：全部文件（跨批次统一浏览和内容查看）
  - `/profile`：个人中心（头像上传、个人信息编辑）
  - `/admin`：管理员全局看板（用户管理、指标管理、模板管理）
- 权限拦截：
  - 前端 React 路由守卫：未登录重定向至 `/login`；非管理员访问 `/admin` 路径时强制跳转到 `/dashboard`。
  - 后端 FastAPI 依赖项 `get_current_user` 验证 Token 并检查 `is_active` 状态；`get_current_admin_user` 对管理端 API 强校验 `is_admin`。
- **应用启动会话恢复**：前端 `App.tsx` 在启动时检测"已认证但无 accessToken"的假登录状态（浏览器重启导致内存 Token 丢失），自动通过 `httpOnly Cookie` 中的 `refresh_token` 静默恢复会话，并调用 `/auth/me` 同步最新的用户信息（含 `is_admin`、`avatar_url`）。
- **忘记密码全链路安全机制**：
  1. 用户输入邮箱请求发送验证码。若邮箱在系统内不存在，返回与成功一致的通用提示（防邮箱枚举攻击）。
  2. 对合法用户，通过安全随机数生成器生成 6 位数字验证码，SHA-256 哈希后存入数据库。调用 SMTP 邮件组件投递。
     - *沙盒降级*：`MAIL_SANDBOX_MODE=true` 时，跳过真实 SMTP 链路，将邮件 HTML 写入 `/logs/mock_mails/`。
  3. 多重请求覆盖熔断：同一邮箱在 10 分钟内多次触发发送时，在单一事务中将该邮箱所有旧码标记为 `is_used = TRUE`，确保仅最新验证码有效。
  4. 密码复杂度底线：长度 8-20 位，必须同时包含大写字母、小写字母、数字及特殊符号（`@$!%*#?&`）中的至少三种。
  5. 提交重置与原子熔断：在单一数据库事务中串联校验：① 验证码是否过期（10 分钟）；② `is_used` 是否为 FALSE；③ 输错次数 `retry_count` 是否 ≥5（超限则该码立即废弃）。校验通过后原子更新密码并置 `is_used = TRUE`。

### 3.2 PDF 批量处理模块

- **明确目标**：实施刚性的文件上传边界校验，构建高效的 Celery 异步处理流式管道，利用 MD5 建立全局去重与结果复用机制。
- **核心业务边界**：
  - **刚性约束限制**：单次批量上传最大文件数 = 10 个。单文件体积必须 ≤ 20MB。若任意一项超限，前端立即弹出 AntD Notification 拒绝提交并中断请求。
  - **上传时可配置项**：用户可输入批次名称（可选），可勾选要提取的指标子集（支持全选/部分勾选/从模板导入/AI 推荐），可点击 AI 推荐按钮让系统分析 PDF 内容后自动推荐指标体系。
  - **文件去重策略（MD5 强匹配）**：
    - 后端接收到 PDF 文件流后，在同步阶段立即计算文件的 MD5 值并检索 `reports` 表。
    - 若命中相同 MD5 且状态为 `success` 的历史记录：① 拒绝磁盘二次写入；② 新报告行的 `stored_path` 直接复制引用历史记录的物理路径；③ 直接复制其 `raw_markdown` 结果并绑定至新批次，实现秒传。秒传判定必须在 POST 同步阶段完成，不得推迟至 Celery 异步管道。
  - **异步流式管道设计**（分发模式）：
    - `process_batch(batch_id)` 收到批次后，遍历所有 pending 报告，逐条分发独立的 Celery 任务 `process_single_report_task`，实现真正的并行处理。
    - **Step 1（解析）**：`parse_pdf()` 使用 PyMuPDF 提取 PDF 前 20 页文本并清洗为 Markdown，写入 `reports.raw_markdown`。超出 20 页部分静默丢弃并在日志记录原始总页数与丢弃页数。
    - **Step 2（提取）**：`extract_metrics_with_ai()` 调用 `_load_batch_metrics()` 从 `batch_metric_relations` 读取该批次绑定的指标子集，通过 `_build_dynamic_prompt()` 动态组装 Prompt。若为默认港股 8 项指标集，使用调优版 `HK_STOCK_SYSTEM_PROMPT`（含章节锚定+数据清洗三原则）；若为自定义指标集，使用通用动态 Prompt。调用硅基流动 API（`deepseek-ai/DeepSeek-V3`，`response_format: json_object`）。
    - **Step 3（持久化）**：`save_metrics()` 读取每项指标的 `expected_type`：NUMERIC 型执行强类型正则清洗（剔除非数字字符）后写入 `metric_value_num`，原始文本写入 `metric_value_raw`；TEXT 型直写 `metric_value_raw`，`metric_value_num` 置 NULL。若大模型偶发缺失某项指标，自动生成兜底容错记录（`metric_value_raw = "AI未输出该指标内容"`）。
    - AI 提取的 `company_name` 自动回填至 `reports.entity_name` 冗余字段。
    - 重试策略：最多 3 次，指数退避 + 随机抖动（30s/60s/120s）；最终失败标记为 `failed` 并记录 `error_message`。
  - **批次状态同步规则**：批次内所有报告 `success` → 批次 `completed`；所有报告 `failed` → 批次 `failed`；部分成功部分失败 → 批次 `partial`。前端对比矩阵和图表分析仅在批次状态为 `completed` 或 `partial` 时可用。

### 3.3 指标提取与对比输出模块

- **明确目标**：基于动态指标配置系统，配合特定的 System Prompt 强约束大模型稳定输出 JSON，并在前端实现基于价格偏离度与资金吞吐量的异常高亮对比矩阵。

#### 3.3.1 动态指标配置系统

- **指标动态配置化**：系统彻底废除对大模型提取指标名称的后端硬编码。大模型"提取什么"完全取决于用户前台的动态配置与勾选。
- **数据存储通用化（EAV 纵表）**：`extracted_metrics` 表采用实体-属性-值模式。当用户自定义新增指标时，后端仅在配置表中增加定义元数据，在提取结果表中增加行记录，严禁在物理表结构上执行 `ALTER TABLE`。
- **双模混合分流机制**：
  1. **默认模式**：用户未勾选指标时，后端自动读取 `is_system=True` 的系统预置指标（8 项港股核心指标）作为默认兜底。
  2. **自定义模式**：用户显式勾选了特定指标集时，后端按用户指定的 `metric_ids` 进行绑定。
  3. **架构统一约束**：经分流补齐后，Celery 异步任务与动态 Prompt 组装层不感知双模差异，统一视作动态批次处理。
- **系统预置指标模板（8 项港股默认指标）**：

| #   | JSON Schema 字段名 (`metric_key`) | 数据类型        | 前端显示名 (`metric_label`) | 业务说明             |
| --- | ---------------------------- | ----------- | ----------------------- | ---------------- |
| 1   | `company_name`               | TEXT        | 公司名称                    | AI 提取后回填 entity_name |
| 2   | `stock_code`                 | TEXT        | 证券代号                    | 规范化为 5 位主板港股代号    |
| 3   | `submission_date`            | TEXT        | 呈交日期                    | 规整为 YYYY-MM-DD    |
| 4   | `repurchase_date`            | TEXT        | 交易日                     | 规整为 YYYY-MM-DD    |
| 5   | `shares_repurchased`         | NUMERIC     | 购回股份数目                  | 数值纯化剔除文本干扰        |
| 6   | `highest_price_paid`         | NUMERIC     | 每股最高购回价                 | 数值纯化剔除文本干扰        |
| 7   | `lowest_price_paid`          | NUMERIC     | 每股最低购回价                 | 数值纯化剔除文本干扰        |
| 8   | `total_consideration`        | NUMERIC     | 付出价格总额                  | 数值纯化剔除文本干扰        |

- **expected_type 与底层存储分流策略**：
  - NUMERIC（数值型）：强类型正则清洗 → `metric_value_num`（DECIMAL 18,4），原始文本 → `metric_value_raw`
  - TEXT（文本型）：直写 `metric_value_raw`，`metric_value_num` 刚性置 NULL
  - 清洗失败或脏数据：`metric_value_num` 置 NULL，异常文本写入 `metric_value_raw`

#### 3.3.2 异常数值检测算法

- **前提条件**：仅当批次绑定的指标集中同时包含 `stock_code`、`highest_price_paid`、`lowest_price_paid`、`shares_repurchased`、`total_consideration` 这 5 项港股核心字段时才启用。
- **小样本边界防护**：仅在当前批次内同一 `stock_code` 的有效报告数 $N \ge 3$ 时生效。$N < 3$ 时自动跳过，防止分母为零或高频误报。
- **三种检测算法**：
  1. **中位数偏离法**：`highest_price_paid` 或 `lowest_price_paid` 偏离同一 stock_code 中位数的 ±5% 时，触发价格异常高亮（浅橘色背景）。
  2. **成交量异动法**：`shares_repurchased` 或 `total_consideration` 超过同批次其他报告均值的 200%（N-1 分母）时，触发成交量异常高亮（浅红色背景 + 高头寸警告图标）。
  3. **IQR 四分位距法 / Z-Score 法**：作为补充检测手段，系统自动选择最合适的算法。
- **三级敏感度**：可配置检测严格程度，分组检测不同 stock_code 的报告。

#### 3.3.3 数据矩阵与导出

- 前端将 MySQL 中的多报告数据拼装成双轴矩阵（行标题 = 指标显示名，列标题 = 文件名/交易日）。
- 异常高亮：值单元格根据异常检测结果渲染背景色 + Tooltip 详细解释检测方法、偏离值、阈值。
- 一键导出：使用前端 `xlsx` 库直接抓取当前 DOM 表格的二维数据流，转换为标准 `.xlsx` 文件下载。
- 动态指标场景下，矩阵列数随用户勾选的指标数动态变化，导出表头联动。

### 3.4 AI 聊天问答系统

- **明确目标**：为用户提供全功能 AI 对话界面（DeepSeek 风格），支持基于报告内容的流式追问、多会话历史管理和全屏沉浸模式。
- **整体架构**：系统包含两个 AI 聊天入口——① 浮动小窗 `ChatWidget`（右下角气泡，可折叠）；② 全功能对话页 `ChatPage`（`/chat` 路由，DeepSeek 风格布局，左侧历史栏 + 右侧对话区）。

#### 3.4.1 全功能对话页（/chat）

- **UI 布局**：左侧 `ChatSidebar` 历史会话列表（可按时间分组：今天/昨天/7天内/30天内/按月、支持折叠）；右侧对话主区域（顶栏 + 消息区 + 输入区）。
- **会话管理**：
  - 新建会话：点击侧边栏「+ 新对话」按钮，后端创建新会话并返回 `session_id`
  - 切换会话：点击历史列表中的会话，加载该会话的全部历史消息
  - 重命名会话：行内编辑（双击或编辑按钮），自动保存
  - 删除会话：Popconfirm 二次确认后级联删除会话及所有消息
  - 清空对话：仅删除会话中全部消息，保留会话本身及其标题
  - 自动标题：发送首条用户消息时，后端自动截取前 30 字符作为会话标题
- **报告绑定**：对话顶栏右侧 Tag 可绑定/切换报告，支持"先选择批次 → 再选择报告"的二级选择流程。绑定报告后 AI 回复将基于该报告的全文内容。默认为"通用模式"（不绑定报告）。
- **全屏模式**：点击全屏按钮，对话区域从当前位置向四周延展为全屏（0.6s CSS transition 动画），ESC 或再次点击退出全屏。
- **快捷提示词**：空状态欢迎页提供 5 条快捷提示（总结核心观点、提取关键指标、分析结论与风险、对比行业数据、分析趋势变化）。
- **Markdown 渲染**：AI 回复使用 `react-markdown + remark-gfm` 渲染（表格、代码块、列表、引用、加粗、链接等），React.memo 优化避免流式输出时历史消息重复渲染。
- **流式性能优化**：`useChatStream` Hook 使用 `requestAnimationFrame` 批量更新 React 状态，多个 SSE chunk 合并到 ~16ms 的渲染帧中，避免高频渲染导致卡顿。

#### 3.4.2 浮动聊天小窗（ChatWidget）

- **UI 表现**：固定悬浮于系统右下角，可折叠/展开。消息气泡：用户消息深蓝色靠右，AI 消息浅灰色靠左。
- **功能**：报告绑定/解绑、快捷提示词、消息清空、流式加载指示器。

#### 3.4.3 聊天核心机制

- **双模态上下文注入**：
  1. **报告绑定模式**：后端读取 `report.raw_markdown` 全文作为 Context 注入 System Prompt
  2. **通用知识模式**：`report_id = null`，System Prompt 切换为基础角色设定，完全基于 DeepSeek-V3 自身知识回答
- **流式传输**：后端统一采用 SSE (Server-Sent Events) 协议，`text/event-stream` 格式逐字推送。API 响应完成后自动保存 AI 回复到数据库。
- **对话流控制**：每次仅携带最新 5 轮（10 条）对话历史。AI 请求上下文通过 `smart_truncate()` 智能截断（保留首尾，中间省略），最大 16000 字符。
- **安全限流**：令牌桶算法，每个用户每分钟最多 10 条消息。Redis 滑动窗口 + 内存兜底。超过限制返回 `429 Too Many Requests`。
- **跨用户数据隔离**：
  - 后端强制校验 `chat_sessions.user_id` 与当前登录用户一致
  - 报告绑定校验：`report.batch.user_id == current_user.id`（管理员豁免）
  - 前端 Zustand：`authStore.setAuth()` 和 `clearAuth()` 均调用 `chatStore.resetAll()` 完全清除聊天数据
  - `session_id` 全局唯一约束冲突时自动生成 UUID 新 ID

### 3.5 多维数据可视化分析

- **明确目标**：为用户提供 6 种图表类型的智能生成与交互分析能力，支持自动模式（系统根据数据特征智能选择图表类型）和手动模式（用户自定义）。
- **技术选型**：ECharts（通过 `echarts-for-react` 封装），按需树摇注册以控制包体积。

#### 3.5.1 图表注册中心（ChartRegistry）

- 提供统一的图表注册/查找/分派机制。所有 6 种图表模块通过副作用导入自注册。
- **autoAssign 自动分派**：根据指标数量和报告数量自动判断每种图表类型的适用性：
  - 折线图（line）：指标 ≥1 且 报告 ≥2
  - 柱状图（bar）：指标 ≥1 且 报告 ≥1
  - 饼图（pie）：指标 =1 且 报告 ≥2
  - 雷达图（radar）：指标 ≥3 且 报告 ≥1
  - 仪表盘（gauge）：指标 ≥1 且 报告 ≥1
  - 热力图（heatmap）：指标 ≥4 或 报告 ≥10
- **双 Y 轴智能构建**：根据量纲检测结果自动分组——金额/价格类指标共用左 Y 轴，数量/比率类指标共用右 Y 轴。量级差异过大时回退为单 Y 轴。

#### 3.5.2 六种图表类型

| 图表类型 | 内部标识 | 中文名 | 适用条件 | 特性 |
|---------|------|------|-----|------|
| 折线图 | line | 折线图 | 指标 ≥1, 报告 ≥2 | 趋势模式（X=fiscal_year）/对比模式；异常点红色标注 |
| 柱状图 | bar | 柱状图 | 指标 ≥1, 报告 ≥1 | 异常柱红色标注；dataZoom 分页 |
| 环形图 | pie | 饼图 | 指标 =1, 报告 ≥2 | Top 8 + "其他"合并；中心显示总计；异常扇区红色边框 |
| 雷达图 | radar | 雷达图 | 指标 ≥3, 报告 ≥1 | 归一化 0-100；polygon 形状；异常点标注 |
| 仪表盘 | gauge | 仪表盘 | 指标 ≥1, 报告 ≥1 | 半圆仪表；颜色分档绿/黄/红；最多 6 个自适应网格 |
| 热力图 | heatmap | 热力图 | 指标 ≥4 或报告 ≥10 | 浅绿→深蓝渐变；异常红边框；dataZoom 纵向分页 |

#### 3.5.3 数据分析中心（/analytics）自动模式

- **单指标卡片**：为每个用户选中的数值指标生成独立折线图（N 张 = 指标数量）
- **多批次对比**：当选中 ≥2 个批次时，追加一张柱状对比图
- **辅助视图**：调用 `autoAssign()` 自动追加符合条件的雷达图、饼图、热力图、仪表盘
- **数据降载**：支持 Top-N 截断、时间聚合（按日/月/季度）、分页控制

#### 3.5.4 工作台图表分析（ChartModal）

- **自动模式**：系统根据数据特征智能选择图表类型，自动获取数据
- **手动模式**：用户选择图表类型（6 种）+ 批次多选 + 指标多选，兼容性校验 + 量纲冲突检测提示

#### 3.5.5 工作台自动看板（AutoChartGrid）

- 每个数值指标生成独立折线图卡片，右上角可一键切换为柱状图
- 附加雷达图（支持指标多选重筛选）和饼图辅助视图

#### 3.5.6 量纲智能检测系统

- **四层推断**：关键词匹配 → 单位推断 → 预置映射（22 个系统指标）→ 兜底 unknown
- **支持 5 种量纲**：数量（count）、金额（amount）、单价（price）、比率（ratio）、未知（unknown）
- **冲突评估**：none / low / high 三级，为双 Y 轴分组和图表颜色绑定提供依据

### 3.6 AI 智能指标推荐

- **明确目标**：利用 AI 分析 PDF 文档内容，自动推荐适用于该文档的指标体系，降低用户手动配置指标的门槛。
- **三种使用模式**：
  1. **上传前模式**（AIMetricRecommenderFromFile）：用户在上传前直接选择 PDF 文件，系统提取文本后分析并推荐指标（纯一次性分析，不写入数据库）
  2. **批次模式**：用户选择已完成处理的批次，系统分析批次中某份报告的内容后推荐指标
  3. **文本描述模式**：用户手动输入文档主题或摘要，系统基于文字描述推荐指标体系
- **流式接收**：支持 SSE 流式接收推荐结果，前端实时展示 AI 的分析过程（含进度条和流式文本）
- **推荐结果处理**：用户可查看/编辑每条推荐指标的显示名称和提取提示词，勾选需要的指标一键创建。可选将推荐结果同时保存为指标模板
- **数值型过滤**：支持仅显示数值型指标的过滤开关
- **上下文截断**：AI 分析时报告内容截断为 12000 字符

### 3.7 指标模板系统

- **明确目标**：允许用户和管理员创建可复用的指标合集模板，一键导入到自定义指标中。
- **模板结构**：每个模板包含名称、描述、分类（category）以及 JSON 格式的指标列表（含 key/label/type/prompt_instruction）。
- **模板分类**：
  - **系统模板**（`is_system=True`）：由管理员创建和维护，对所有用户可见
  - **用户模板**：用户自行创建，仅自己可见
- **模板导入**：用户在指标选择页可浏览模板列表（系统模板组 + 用户模板组），选择模板后预览指标详情，确认后一键导入到自己的 `metric_definitions` 中。
- **管理端模板管理**：管理员可创建/编辑/删除系统合集模板，支持一键全部启用/禁用，级联同步到关联的系统指标。

### 3.8 个人中心与头像管理

- **明确目标**：为用户提供个人信息的自助管理入口，包括头像上传、昵称编辑和邮箱修改。
- **具体功能**：
  - **头像上传**：支持 JPG/PNG 格式，单文件限制 2MB。存储于 `uploads/avatars/` 目录，通过 `/uploads` 静态挂载访问。上传新头像时自动删除旧头像文件。前端顶部 Header 和聊天消息头像同步显示最新头像。
  - **昵称编辑**：直接修改，无需验证密码。
  - **邮箱修改**：两步验证码流程。① 输入新邮箱 → 发送验证码；② 输入验证码 → 更新邮箱。新邮箱不可被其他用户占用。
- **前端路由**：`/profile`（需登录），入口位于右上角用户下拉菜单。

### 3.9 后台管理

- **明确目标**：为管理员提供全局数据看板、用户管理、系统指标管理、合集模板管理和审计能力。
- **功能清单**：

#### 3.9.1 用户管理

- 分页展示所有已注册用户（邮箱、昵称、角色、状态、注册时间、处理批次数）
- 搜索：按邮箱或昵称模糊搜索
- 筛选：按激活/冻结状态筛选
- 排序：按注册时间升降序
- 启用/禁用：切换用户 `is_active` 状态（不能禁用自己的管理员账号）
- 审计查看：穿透查看指定用户的所有上传批次（含批次状态、文件数、创建时间）

#### 3.9.2 系统指标管理

- 查看所有系统指标定义（含已禁用）
- 创建/编辑/删除系统指标定义
- 快速切换指标启用/禁用状态

#### 3.9.3 合集模板管理

- 查看所有合集模板（含已禁用）
- 创建/编辑/删除合集模板（模板内含指标 JSON 列表）
- 快速切换模板启用/禁用（级联同步关联系统指标）
- 一键全部启用/禁用所有系统模板

#### 3.9.4 报告穿透查看

- 管理员可查看任意报告的 `raw_markdown` 原文及大模型提取的原始 JSON 结构体

### 3.10 批次与报告管理

#### 3.10.1 上传管理

- 上传时自定义批次名称
- 勾选指标子集（支持全选、部分勾选、从模板导入、AI 推荐）
- 上传进度跟踪：`ProcessingProgressOverlay` 实时显示活跃批次的处理进度

#### 3.10.2 批次管理

- 批次列表：分页展示，每行显示名称、状态（含图标）、进度、绑定指标标签（可展开/收起）、创建时间
- 行内重命名：双击批次名称触发编辑，回车或失焦自动保存
- 删除批次：级联删除关联报告、指标数据，并从磁盘物理删除 PDF 文件
- 一键删除所有批次
- 更新批次指标绑定（`PUT /batches/{batch_id}/metrics`）

#### 3.10.3 报告管理

- 跨批次文件列表（`/files` 页面）：按文件名搜索，按批次/状态筛选
- 报告内容查看：Markdown 原文渲染 + 内嵌 PDF 预览（iframe）
- 报告重命名：修改 `original_filename` 字段
- 报告删除：删除单个报告及关联指标和磁盘文件。若批次内仅剩此报告，同时删除整个批次。

### 3.11 全局文件浏览（/files）

- **明确目标**：提供跨批次的统一文件浏览入口，方便用户快速查找和查看所有上传的 PDF 文件。
- **功能**：搜索（文件名模糊匹配）、筛选（按批次、按处理状态）、分页浏览
- **报告详情**：点击文件可查看 Markdown 提取内容 + 内嵌 PDF 预览（iframe）

## 5. 数据库设计（MySQL 8.0.30 物理表结构）

### 5.2 物理表 DDL

```sql
CREATE DATABASE IF NOT EXISTS deepreader_db CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

-- 1. 用户表
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,
    email VARCHAR(255) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    nickname VARCHAR(100) NOT NULL,
    avatar_url VARCHAR(500) NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NULL ON UPDATE CURRENT_TIMESTAMP,
    is_admin BOOLEAN DEFAULT FALSE,
    is_active BOOLEAN DEFAULT TRUE
) ENGINE=InnoDB;

-- 2. 忘记密码验证码安全审计表
CREATE TABLE password_reset_codes (
    id INT AUTO_INCREMENT PRIMARY KEY,
    email VARCHAR(255) NOT NULL,
    code_hash VARCHAR(64) NOT NULL,
    retry_count INT DEFAULT 0,
    is_used BOOLEAN DEFAULT FALSE,
    expired_at TIMESTAMP NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_email_lookup (email, is_used, expired_at)
) ENGINE=InnoDB;

-- 3. 用户自定义指标配置表
CREATE TABLE metric_definitions (
    id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT NOT NULL,
    metric_key VARCHAR(100) NOT NULL,
    metric_label VARCHAR(100) NOT NULL,
    expected_type ENUM('NUMERIC', 'TEXT') DEFAULT 'NUMERIC',
    prompt_instruction VARCHAR(500) NULL,
    is_system BOOLEAN DEFAULT FALSE,
    is_active BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NULL ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uidx_user_key (user_id, metric_key),
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
) ENGINE=InnoDB;

-- 4. 指标合集模板表
CREATE TABLE metric_templates (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    description VARCHAR(500) NULL,
    category VARCHAR(50) NULL,
    is_system BOOLEAN DEFAULT FALSE,
    is_active BOOLEAN DEFAULT TRUE,
    user_id INT NULL,
    metrics JSON NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NULL ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
) ENGINE=InnoDB;

-- 5. 上传批次表
CREATE TABLE upload_batches (
    id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT NOT NULL,
    batch_name VARCHAR(255) NOT NULL,
    status ENUM('pending', 'processing', 'completed', 'failed', 'partial') DEFAULT 'pending',
    total_files INT DEFAULT 0,
    processed_files INT DEFAULT 0,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NULL ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    INDEX idx_user_batches (user_id)
) ENGINE=InnoDB;

-- 6. 批次与自定义指标关联表
CREATE TABLE batch_metric_relations (
    id INT AUTO_INCREMENT PRIMARY KEY,
    batch_id INT NOT NULL,
    metric_def_id INT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uidx_batch_metric (batch_id, metric_def_id),
    FOREIGN KEY (batch_id) REFERENCES upload_batches(id) ON DELETE CASCADE,
    FOREIGN KEY (metric_def_id) REFERENCES metric_definitions(id) ON DELETE CASCADE
) ENGINE=InnoDB;

-- 7. 单个报告表
CREATE TABLE reports (
    id INT AUTO_INCREMENT PRIMARY KEY,
    batch_id INT NOT NULL,
    original_filename VARCHAR(255) NOT NULL,
    stored_path VARCHAR(500) NOT NULL,
    pdf_md5 VARCHAR(32) NOT NULL,
    file_size INT NOT NULL,
    raw_markdown LONGTEXT NULL,
    entity_name VARCHAR(255) NULL,
    parse_status ENUM('pending', 'parsing', 'extracting', 'success', 'failed') DEFAULT 'pending',
    error_message TEXT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NULL ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (batch_id) REFERENCES upload_batches(id) ON DELETE CASCADE,
    INDEX idx_pdf_md5 (pdf_md5)
) ENGINE=InnoDB;

-- 8. 提取结果表（EAV 纵表）
CREATE TABLE extracted_metrics (
    id INT AUTO_INCREMENT PRIMARY KEY,
    report_id INT NOT NULL,
    metric_name VARCHAR(100) NOT NULL,
    metric_display_name VARCHAR(100) NOT NULL,
    metric_value_num DECIMAL(18, 4) NULL,
    metric_value_raw VARCHAR(500) NULL,
    fiscal_year VARCHAR(20) NULL,
    unit VARCHAR(50) NULL,
    confidence FLOAT DEFAULT 1.0,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NULL ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (report_id) REFERENCES reports(id) ON DELETE CASCADE,
    INDEX idx_report_metric (report_id)
) ENGINE=InnoDB;

-- 9. 聊天会话表
CREATE TABLE chat_sessions (
    id INT AUTO_INCREMENT PRIMARY KEY,
    session_id VARCHAR(100) NOT NULL UNIQUE,
    user_id INT NOT NULL,
    report_id INT NULL,
    title VARCHAR(200) NULL,
    is_active BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NULL ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY (report_id) REFERENCES reports(id) ON DELETE SET NULL,
    INDEX idx_session_lookup (session_id),
    INDEX idx_user_lookup (user_id)
) ENGINE=InnoDB;

-- 10. 聊天消息表
CREATE TABLE chat_messages (
    id INT AUTO_INCREMENT PRIMARY KEY,
    session_id INT NOT NULL,
    role VARCHAR(20) NOT NULL,
    content TEXT NOT NULL,
    model_used VARCHAR(50) DEFAULT 'deepseek-ai/DeepSeek-V3',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (session_id) REFERENCES chat_sessions(id) ON DELETE CASCADE,
    INDEX idx_session_history (session_id)
) ENGINE=InnoDB;
```

## 8. 日志与监控策略

| 层级 | 日志工具 | 输出目标 | 级别 |
|------|----------|----------|------|
| FastAPI 请求日志 | Python `logging` | stdout + `logs/app.log` | INFO（生产）/ DEBUG（开发） |
| Celery 任务日志 | Celery 内置日志 | stdout（worker 输出） | INFO |
| AI API 调用错误 | Python `logging` | stdout | ERROR |
| 安全敏感操作 | Python `logging` | `logs/security.log` | WARNING（登录失败、验证码爆破、权限越权） |

- 审计日志保留 90 天，应用日志保留 30 天
- 日志格式：`[时间戳] [级别] [模块] [用户ID] 消息内容`
