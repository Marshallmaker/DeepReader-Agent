# 第4章 系统概要设计

本章从整体架构、功能模块划分和数据库设计三个层面，给出系统的主体骨架。

## 4.1 系统架构设计

DeepReader Agent采用前后端分离的四层架构，自顶向下依次为表现层、服务层、中间件层和数据层。图4.1展示了各层之间的调用关系。

> **【图4.1 放置位置说明】**
> 此处放置 **系统架构图**。采用分层架构图的画法，自上而下四层：
>
> 第一层：表现层。包含浏览器中运行的React单页应用，标注Vite构建工具和Ant Design组件库。表现层通过HTTP和SSE协议与服务层通信。
>
> 第二层：服务层。包含FastAPI应用服务器，标注8个路由模块，分别是auth、files、batches、chat、admin、metrics、templates和visualization。服务层通过Redis发送任务到Celery工作进程，通过SQLAlchemy操作MySQL。
>
> 第三层：中间件层。左侧为Celery工作进程进程，标注4并发线程池和PDF处理管道；右侧为Redis实例，标注消息代理和结果后端两个角色。
>
> 第四层：数据层。包含MySQL 8.0数据库，标注10张核心表；以及文件系统中的uploads目录，存储用户上传的PDF和头像文件。
>
> 层与层之间用箭头标明数据流向和通信协议。建议使用draw.io的分层框图模板绘制。

图4.1已经标出了各层的关键组件和通信关系，这里补充三点图中不易体现的设计决策。一是服务层启动时自动执行init_db确保数据库表结构就绪，统一异常处理器将所有后端错误转换为中文响应。二是中间件层的Redis除了图中标注的消息代理和结果后端两个角色外，还承担JWT令牌黑名单的存储，利用SETEX命令的自动过期能力实现登出令牌的自动失效。三是数据层的文件存储按年月组织子目录，前端通过FastAPI挂载的静态文件路由直接访问，无需额外的文件服务。

## 4.2 系统功能模块设计

系统按照用户角色划分功能模块。普通用户和管理员各自拥有独立的功能视图，管理员的功能视图在普通用户的基础上扩展了管理类模块。图4.2展示了两类用户的模块归属关系。

> **【图4.2 放置位置说明】**
> 此处放置 **系统功能模块图**，按用户角色分组展示。图中分为左右两大区域：
>
> 左侧为普通用户模块区，包含6个模块：
> · 用户认证：注册、登录、忘记密码、个人信息
> · 文件与批次管理：上传PDF、文件列表、批次列表、指标绑定、处理进度
> · 数据分析：对比矩阵、图表分析、自动图表网格
> · AI对话：会话管理、报告对话、通用问答
> · 指标库：自定义指标、AI推荐指标、模板导入
> · 报告查看：PDF预览、Markdown内容查看
>
> 右侧为管理员专属模块区，包含3个模块：
> · 用户管理：用户列表、状态启停、批次审计
> · 系统指标管理：指标CRUD、启用禁用
> · 系统模板管理：模板CRUD、启用禁用、一键切换
>
> 6个普通用户模块用实线框表示，3个管理员模块用虚线框叠加在普通用户模块上方，表示扩展关系。建议使用ProcessOn或Visio绘制。

普通用户拥有6个功能模块，覆盖从文件上传到分析结果获取的完整流程。管理员在普通用户基础上扩展了3个管理模块，两类用户的模块归属关系见图4.2。

## 4.3 数据库设计

### 4.3.1 概念结构设计

数据库的概念模型围绕10个实体构建，实体之间的关系反映了系统的业务规则。图4.3以ER图形式展示了核心实体及其关联。

> **【图4.3 放置位置说明】**
> 此处放置 **数据库ER图**。图中包含以下主要实体及关系：
>
> 核心实体：用户、上传批次、报告、提取指标、指标定义、批次指标关联、指标模板、对话会话、对话消息、密码重置码。
>
> 主要关系：
> · 用户与上传批次：一对多，一个用户可创建多个批次
> · 上传批次与报告：一对多，一个批次包含多份报告
> · 报告与提取指标：一对多，一份报告包含多条指标
> · 用户与指标定义：一对多，用户可自定义多个指标
> · 指标定义与批次指标关联：一对多，关联到批次
> · 上传批次与批次指标关联：一对多，绑定指标集
> · 用户与对话会话：一对多，用户可创建多个对话会话
> · 对话会话与对话消息：一对多，一个会话包含多条消息
> · 报告与对话会话：多对多，通过绑定字段关联
> · 用户与指标模板：一对多，用户可创建多个模板
>
> 建议使用MySQL Workbench或draw.io的ER图模板绘制，标注各实体的主键和关键外键。

10个实体及其关系详见图4.3。其中，指标定义和批次指标关联共同构成了动态指标系统的核心：不同批次可以绑定不同的指标子集，一个批次只提取用户关心的指标，而不是每次都处理全部指标定义。对话会话首次创建时，系统自动截取用户消息前30个字符作为默认标题。

### 4.3.2 逻辑结构设计

概念模型映射到MySQL后，形成10张物理表。以下列出每张表的核心字段，完整的建表SQL见附录。

**users 用户表**


| 字段            | 类型           | 说明           |
| ------------- | ------------ | ------------ |
| id            | INT          | 主键，自增        |
| email         | VARCHAR(255) | 邮箱，唯一索引      |
| password_hash | VARCHAR(255) | bcrypt哈希后的密码 |
| nickname      | VARCHAR(100) | 用户昵称         |
| is_admin      | TINYINT      | 是否管理员，0或1    |
| is_active     | TINYINT      | 账号是否启用，0或1   |
| avatar_url    | VARCHAR(500) | 头像文件路径       |
| created_at    | DATETIME     | 创建时间         |
| updated_at    | DATETIME     | 更新时间         |


**password_reset_codes 密码重置码表**


| 字段          | 类型           | 说明            |
| ----------- | ------------ | ------------- |
| id          | INT          | 主键，自增         |
| email       | VARCHAR(255) | 目标邮箱          |
| code_hash   | VARCHAR(255) | SHA256哈希后的验证码 |
| retry_count | INT          | 重试次数          |
| is_used     | TINYINT      | 是否已被使用，0或1    |
| expires_at  | DATETIME     | 过期时间          |


**upload_batches 上传批次表**


| 字段              | 类型           | 说明                                  |
| --------------- | ------------ | ----------------------------------- |
| id              | INT          | 主键，自增                               |
| user_id         | INT          | 外键，关联users表                         |
| batch_name      | VARCHAR(200) | 批次名称                                |
| status          | ENUM         | 状态：pending、completed、failed、partial |
| total_files     | INT          | 批次内文件总数                             |
| processed_files | INT          | 已完成处理的文件数                           |
| created_at      | DATETIME     | 创建时间                                |
| updated_at      | DATETIME     | 更新时间                                |


**reports 报告表**


| 字段                | 类型           | 说明                                     |
| ----------------- | ------------ | -------------------------------------- |
| id                | INT          | 主键，自增                                  |
| batch_id          | INT          | 外键，关联upload_batches表                   |
| original_filename | VARCHAR(500) | 原始文件名                                  |
| stored_path       | VARCHAR(500) | 服务器存储路径                                |
| pdf_md5           | VARCHAR(64)  | 文件MD5哈希，用于去重                           |
| file_size         | BIGINT       | 文件大小，字节数                               |
| status            | ENUM         | 状态：pending、processing、completed、failed |

| updated_at        | DATETIME     | 更新时间                                   |


**extracted_metrics 提取指标表**


| 字段                  | 类型            | 说明            |
