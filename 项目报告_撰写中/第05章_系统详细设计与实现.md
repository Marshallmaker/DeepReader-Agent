## 5.1 用户认证模块

用户认证模块处理注册、登录、令牌刷新和密码重置四类操作。本系统采用JWT双令牌认证方案：登录时后端签发两个令牌，访问令牌用于接口鉴权，有效期15分钟，返回在JSON响应体中；刷新令牌用于续期，有效期7天，以httpOnly Cookie形式存储在浏览器中。用户勾选"记住我"时Cookie持久化保存，不勾选则为会话级。

> **【图5.1 放置位置说明】**
> 此处放置 **登录与令牌刷新流程图**。图中包含三条流程线：
>
> 1. 登录流程：用户提交邮箱和密码，后端验证，签发访问令牌和刷新令牌，访问令牌返回JSON body，刷新令牌写入httpOnly Cookie
> 2. 正常请求流程：前端携带访问令牌发起请求，后端JWT校验，通过则返回数据
> 3. 令牌过期刷新流程：请求返回401，Axios拦截器捕获，全局互斥调用refresh接口，Cookie自动携带刷新令牌，后端签发新访问令牌，重试原请求
>
> 建议使用泳道图，三个角色分别为"浏览器""前端应用""后端服务"。

> **【图5.2 放置位置说明】**
> 此处放置 **登录页面效果图**和**注册页面效果图**。登录页左侧为表单区，含邮箱、密码、记住我复选框、登录按钮和忘记密码链接，右侧为注册入口。注册页为两步流程：第一步输入邮箱并获取验证码，第二步设置密码和昵称。

登录接口的核心逻辑如下：用户提交邮箱和密码，后端查询users表，使用bcrypt比对密码哈希。验证通过后生成两个JWT令牌，访问令牌的载荷包含user_id、email、is_admin和type字段，刷新令牌的type标记为refresh。

```python
# JWT 令牌签发
def create_access_token(data: dict) -> str:
    to_encode = data.copy()
    expire = datetime.utcnow() + timedelta(
        minutes=settings.ACCESS_TOKEN_EXPIRE_MINUTES)
    to_encode.update({"exp": expire, "type": "access"})
    return jwt.encode(
        to_encode, settings.SECRET_KEY, algorithm=settings.ALGORITHM)
```

令牌黑名单机制处理用户主动登出。登出时将访问令牌的唯一标识存入Redis，过期时间设为令牌剩余有效期。后续请求中JWT依赖注入函数检查该令牌是否在黑名单中。Redis不可用时自动降级为内存集合存储。

## 5.2 PDF上传与AI指标提取模块

本模块的任务链路是文件接收、校验去重、记录创建、Celery任务分发、PDF解析、AI指标提取和数据入库七个步骤。

> **【图5.3 放置位置说明】**
> 此处放置 **PDF处理管道流程图**。图中展示从用户上传到数据入库的完整链路：
> 用户上传，文件校验，MD5计算，去重检查，命中则复用已有记录，创建UploadBatch和Report记录，绑定指标集到BatchMetricRelation，调用process_batch.delay分发Celery任务，前端返回处理中状态
>
> Celery工作进程内部流程：接收process_batch，遍历批次内报告，为每份报告生成process_single_report_task子任务，PyMuPDF解析文本，读取批次绑定的指标定义，动态组装提示词，调用DeepSeek-V3 API，解析JSON响应，按数值型和文本型分流写入extracted_metrics表，更新报告状态，所有子任务完成后联动更新批次状态
>
> 建议使用流程图加泳道，区分"前端""FastAPI后端""Celery工作进程""DeepSeek API"四个角色。

文件校验和去重在上传接口中同步完成。校验通过的文件先计算MD5哈希，然后在reports表中查找相同哈希的已完成记录。命中后直接复用已有记录的原始内容内容，创建新的批次绑定关系，跳过后续AI处理。

```python
# MD5去重逻辑
existing = db.query(Report).filter(
    Report.pdf_md5 == file_md5,
    Report.status == "completed"
).first()

if existing and existing.raw_markdown:
    new_report = Report(
        batch_id=batch.id,

```

Celery分发模式下process_batch只负责遍历和分发，每份报告的处理由独立的子任务完成。每个子任务有独立的失败重试逻辑，指数退避策略下重试间隔递增，依次为10秒、100秒和1000秒，最多重试3次。

动态提示词的组装方式为：从batch_metric_relations表读取当前批次绑定的指标定义列表，遍历每个指标的metric_key、metric_label和prompt_instruction，拼接成发给大模型的系统提示词。如果指标集恰好是港股默认8项，使用经过调优的HK_STOCK_SYSTEM_PROMPT；否则使用通用的动态组装模板。

```python
# 动态提示词组装
def _build_dynamic_prompt(metric_defs: list) -> str:
    if _is_default_hk_stock_set(metric_defs):
        return HK_STOCK_SYSTEM_PROMPT
    
    metric_lines = []
    for md in metric_defs:
        line = f"- {md.metric_key}: {md.metric_label}"
        if md.prompt_instruction:
            line += f"  {md.prompt_instruction}"
        metric_lines.append(line)
    
    return GENERAL_SYSTEM_PROMPT_TEMPLATE.format(
        metrics="\n".join(metric_lines)
    )
```

AI返回的JSON对象中，每个字段的值由save_metrics函数按类型分流处理。expected_type为数值型的指标，通过正则清洗去除非数字字符后转为Decimal写入metric_value_num；expected_type为文本型的指标，保留原始字符串写入metric_value_raw。fiscal_year字段统一使用报告呈交日期作为锚点。

## 5.3 数据可视化与对比分析模块

数据可视化模块的核心是一个图表注册中心ChartRegistry。系统中所有可用的图表类型都在注册中心登记，每类图表实现了autoAssign自动分派逻辑：根据传入的指标数量、数据类型和时间跨度，自动判断最适合的图表类型。系统内置折线图、柱状图、环形图、雷达图、仪表盘和热力图共6种图表。

> **【图5.5 放置位置说明】**
> 此处放置 **图表分析页面效果图**。建议截取Analytics页面，展示：
>
> 1. AutoChartGrid自动图表网格视图：多张指标卡片以网格排列，每张卡片内为独立折线图，右上角可切换柱状图，左下角有雷达图或环形图辅助视图入口
> 2. ChartModal手动分析弹窗：左侧为批次和指标多选列表，右侧为生成的图表预览，顶部有图表类型切换选项卡，底部有量纲冲突提示信息

ChartRegistry将指标按量纲自动分为两组：百分比类指标映射到左侧Y轴，绝对值类指标映射到右侧Y轴。分组依据来自DimensionDetector的量纲检测结果，检测逻辑分为四层：关键词识别、单位匹配、预置映射表和兜底默认分组。如果同一组内存在量纲冲突，系统在图表区域显示警告提示。

```typescript
function autoAssign(metrics: MetricInfo[]): ChartType {
  if (metrics.length === 1) {
    const m = metrics[0];
    if (m.dataPoints.length === 1) return 'gauge';
    if (m.dataPoints.length <= 12) return 'line';
    return 'line';
  }
  if (metrics.length <= 3) return 'radar';
  if (metrics.length <= 10) return 'bar';
  return 'heatmap';
}
```

对比矩阵在批次详情页中以表格形式呈现，每行为一份报告，每列为一项指标。异常检测在渲染矩阵时执行：对每组共享同一股票代码的报告运行统计检测，方法根据样本数和分布特征自动选择，异常单元格以浅橘色背景标出，鼠标悬停时显示偏离方向和程度。

## 5.4 AI智能对话模块

AI对话模块支持两种模式：报告绑定模式下，系统将选定报告的原始内容全文作为上下文注入提示词；通用知识模式下无报告绑定，用户自由提问。两种模式共用同一套会话管理。

> **【图5.7 放置位置说明】**
> 此处放置 **AI对话页面效果图**。建议截取ChatPage全屏模式：
>
> 1. 左侧为ChatSidebar会话列表，按今天、昨天、7天内、30天内和更早分组，每个会话项显示标题和最后活跃时间，支持行内重命名和删除
> 2. 右侧为对话区域，顶部显示会话标题和报告绑定状态，中间为消息流，用户消息右对齐蓝底，AI回复左对齐白底并以Markdown渲染，底部为输入框和发送按钮
> 3. 右下角附一张ChatWidget浮动小窗效果图，展示可折叠的悬浮聊天组件

对话传输基于SSE协议，后端通过FastAPI的StreamingResponse逐步推送AI生成的文本片段。前端使用requestAnimationFrame批量更新渲染，多个SSE数据块合并到一帧内处理，避免每个数据块触发一次React重渲染。

```typescript
const buffer = useRef<string[]>([]);

function onChunk(text: string) {
  buffer.current.push(text);
  if (!rafRef.current) {
    rafRef.current = requestAnimationFrame(() => {
      setContent(prev => prev + buffer.current.join(''));
      buffer.current = [];
      rafRef.current = null;
    });
  }
}
```

上下文窗口管理采用滑动策略，每次向AI发送消息时仅携带最近5轮共10条对话历史，更早的消息被裁切。首次对话时系统自动截取用户第一条消息的前30个字符作为会话默认标题。会话列表按时间分组显示，支持创建、重命名、删除和清空消息操作。
